# Assumptions & Design Decisions

This document captures all assumptions made during the implementation of the IPN P2P Payment Integration Challenge, including the reasoning behind each decision and how a production system would differ.

---

## Error Simulation

### ERR005 — Insufficient Funds

**Spec says:** Error code ERR005 exists with HTTP 402 and description "Insufficient funds."

**What's missing:** No trigger condition is defined — the spec doesn't say when to return this error.

**Our assumption:** Amounts exceeding **10,000 NAD** trigger ERR005.

**Why this approach:**
- In a real system, this would check the sender's account balance against a core banking system. Since database persistence is out of scope, we simulate with a threshold.
- **10,000 NAD** was chosen because:
  - It's a round number that's easy for reviewers to test
  - It's realistic — many payment systems enforce per-transaction limits
  - It's deterministic and predictable, unlike random simulation
  - It demonstrates understanding that payment systems have limits

**Production alternative:** Query the core banking system for the sender's available balance and compare against the requested amount.

### ERR006 — Internal Processing Error

**Spec says:** Error code ERR006 exists with HTTP 500 and description "Internal processing error."

**What's missing:** No trigger condition — real internal errors are by nature unpredictable.

**Our assumption:** ERR006 is triggered deterministically when the `clientReference` ends with the suffix `-ERR`.

**Why this approach:**
- Random failures (e.g., 5% probability) make automated testing unreliable
- A deterministic trigger allows reviewers to test error handling on demand
- It demonstrates understanding that systems need testable error paths

**How to test:** Submit a payment with `clientReference` ending in `-ERR` (e.g., `REF-20260315-abc12-ERR`).

**Production alternative:** ERR006 would occur naturally from database timeouts, network failures, or unhandled exceptions. The API would catch these and return a generic 500 response.

## Validation Rules

### Account Numbers as Strings

**Spec says:** `senderAccountNumber` and `receiverAccountNumber` are type `string` with validation rule "Numeric only."

**Our interpretation:** Account numbers are stored and validated as strings, never converted to numbers.

**Why:** Real bank account numbers can have leading zeros (e.g., `0987654321` from the spec's sample request). Converting to `number` type would silently drop the leading zero, corrupting the account identifier. Account numbers are identifiers, not quantities — you never perform arithmetic on them.

### Self-Transfer Prevention

**Spec says:** Nothing — the spec does not mention whether sender and receiver can be the same.

**Our assumption:** `senderAccountNumber === receiverAccountNumber` returns ERR002 ("Invalid account number format").

**Why:** Real payment systems prevent or flag self-transfers. Including this check demonstrates awareness of real-world payment system constraints beyond what's written in a spec.

### No Maximum Account Number Length

**Spec says:** Minimum length 10 digits. No maximum specified.

**Our assumption:** No maximum is enforced — any length >= 10 with only digits is accepted.

**Production alternative:** Real systems would have a maximum (e.g., 20 digits for IBANs, or bank-specific limits). This would be defined by the IPN integration specification.

### Amount Precision

**Spec says:** `amount` is type `number` and "must be greater than 0."

**Our handling:** Amounts are rounded to 2 decimal places during validation to handle IEEE 754 floating-point precision issues (e.g., `0.1 + 0.2 = 0.30000000000000004` in JavaScript).

**Production alternative:** Real financial systems use integer representations (amount in cents) or dedicated Decimal libraries to avoid floating-point issues entirely.

## UI Decisions

### Currency Field (Read-Only)

**Spec says:** `currency` is required and must equal "NAD."

**Our decision:** The currency field is pre-filled with "NAD" and displayed as read-only. Users cannot change it.

**Why:** Since only one value is valid, allowing users to type a different currency and then showing an error creates unnecessary friction. A read-only field communicates the constraint clearly.

### Client Reference (Auto-Generated)

**Spec says:** `clientReference` is a "unique transaction reference generated by the client."

**Our decision:** The client reference is auto-generated on each form submission using the format `REF-YYYYMMDD-XXXXX` (date + 5-character random hex). It is displayed to the user but is not editable.

**Why:** The spec says it's "generated by the client" (the application), not "entered by the user." Auto-generation prevents human errors and ensures the format is consistent. The random hex component provides stateless uniqueness without needing a database sequence.

**Production alternative:** Use ULIDs or UUIDs for guaranteed uniqueness in distributed systems, possibly with a database sequence for human-readable sequential references.

## API Behaviour

### Duplicate Client Reference Detection

**Spec says:** `clientReference` must be "unique per transaction."

**Our implementation:** The server maintains an in-memory `Set` of previously seen client references. If a reference is seen twice, ERR001 ("Missing required field") is returned with a custom message indicating duplication.

**Limitation:** The Set resets when the server restarts. This is acceptable for a mock implementation.

**Production alternative:** Store client references in a database with a unique constraint. This enables persistent duplicate detection and supports idempotent retry patterns — a critical feature in payment systems where network timeouts can cause duplicate submissions.

### Transaction ID Format

**Spec says:** `transactionId` is a "unique transaction identifier" generated by the server.

**Our format:** `TXN-YYYYMMDDHHMMSS-NNNN` (timestamp + incrementing 4-digit counter).

**Why:** The format mirrors the spec's sample (`TXN202603060001`) but adds dashes for readability. The counter ensures uniqueness within the same second.

**Production alternative:** Distributed ID generators (Twitter Snowflake, ULID) that guarantee uniqueness across multiple server instances.

### Validation Order

Our API validates in this order:
1. JSON parsing (malformed body → ERR006)
2. Schema validation via Zod (→ ERR001, ERR002, ERR003, ERR004)
3. Duplicate clientReference check (→ ERR001)
4. Business logic: insufficient funds (→ ERR005)
5. Error simulation: -ERR suffix (→ ERR006)
6. Success

**Why this order:** Cheapest checks run first. Parsing and schema validation are fast. Business logic checks (which in production would require database queries) run later. This mirrors real payment gateway design.

## Technology Choices

### Next.js with App Router

**Why:** Full-stack in a single project. The API route (`/api/p2p-payment`) cleanly simulates the IPN backend, and the React frontend handles the form and result display. App Router is the current standard for new Next.js applications.

### Zod for Validation (Client + Server)

**Why:** The same validation schema is shared between client-side form validation (for instant UX feedback) and server-side API validation (for security — never trust client data). This "defense in depth" approach ensures the rules can never drift apart between client and server.

### TypeScript Strict Mode

**Why:** Financial systems require maximum type safety. Strict mode catches potential runtime errors at compile time — especially important when handling money and account identifiers.
